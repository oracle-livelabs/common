Step 1 / WMS gate

Submit Workshop Request

Create the WMS record first, fill it like a reviewer will read it, and understand the status path before you spend real time building the workshop.

  • Output: approved request
  • Time: 5 to 10 minutes
  • Leads to: repository work
WMS is not just intake. The same record follows the workshop through approval, Self Quality Assurance, publishing, and Quarterly Quality Assurance.
Start in WMS

Submit the request before development starts

The first reviewer should be able to tell what the learner will build, who owns the workshop, and what special delivery needs exist without asking follow-up questions.

  1. Connect to Corporate VPN, open Oracle Workshop Management System, and click Submit a New Workshop Request.
  2. Complete Workshop Basic Information thoroughly. The fields that matter most at this stage are Stakeholder, Workshop Council, Workshop Owner Group, Workshop Abstract, Workshop Outline, and Workshop Prerequisites.
  3. Open Tags and set the required tag groups before you click Create: one Level, at least one Role, at least one Focus Area, and at least one Product.
  4. Use Go to Market - Social while the workshop is still early if blog, social, or video details will be needed later in the publish flow.
  5. Do not treat GitHub work as the main source of truth until the request has been reviewed and moved out of the initial council gate.
Example completion

Fill the request with concrete outcome-driven detail

Use examples like these as the quality bar. The exact stakeholder or council name changes by workshop, but the level of detail should not.

Workshop Title

Build and publish an Oracle LiveLabs workshop from WMS to GitHub Pages

Lead with the learner outcome, not only the product name.

Workshop Abstract

Authors learn how to request a workshop in WMS, prepare GitHub Desktop and Visual Studio Code, build from the LiveLabs sample structure, complete Self Quality Assurance, and request publishing.

A council reviewer should understand the end-to-end learner goal after reading two or three sentences.

Workshop Outline
  1. Submit and track the WMS request.
  2. Set up GitHub and preview tooling.
  3. Build the workshop structure and labs.
  4. Run Self Quality Assurance, fix pull request issues, and publish.

Outline the build flow in the same order the learner or author will execute it.

Workshop Prerequisites

Oracle VPN access, GitHub account tied to @oracle.com, GitHub Desktop, Visual Studio Code, Live Server, and permission to work in the target oracle-livelabs repository.

If a prerequisite blocks setup or review later, surface it now.

Stakeholder / Council / Owner Group

Choose the named stakeholder who will verify the workshop, the council aligned to the production repository, and the team that will own maintenance after publish.

Do not leave these as defaults or temporary contributors.

Required Tags

Use the actual WMS tags for the workshop's level, audience role, focus area, and primary product.

  • Level = Beginner
  • Role = Developer
  • Focus Area = the main solution area
  • Product = the service being taught

Tags are mandatory search and routing metadata, not optional cleanup.

Status flow

Understand the handoff path before you build

The workshop is not publish-ready just because the request exists. WMS status controls when heavy work starts, when Quality Assurance is valid, and when publishing can move ahead.

Submitted

The council sees the request and starts its first review pass.

More Info Needed or Approved

Answer council questions if they need more detail. Wait for approval before heavy build work whenever possible.

In Development

Use this after real authoring starts in GitHub and the preview path exists.

Self Quality Assurance to Self Quality Assurance Complete

Run end-to-end testing, save the checklist, and certify the handoff only after the workshop is stable.

Completed

Stakeholders have verified the workshop and it is ready for final publish handling.

Quarterly Quality Assurance

Published workshops cycle back through Quality Assurance later. If Quality Assurance is missed, the production entry can be disabled.

Before you move on

Step 1 exit criteria

  • The title, abstract, outline, and prerequisites tell a reviewer exactly what the workshop will accomplish.
  • Stakeholder, council, owner group, and required tags are already correct in WMS.
  • You know whether the request is still waiting for council review or has been approved to move into development.
  • You understand that later Self Quality Assurance, stakeholder review, and Quarterly Quality Assurance all reuse this same WMS record.
Watch for

Common first-step mistakes

  • Writing a vague abstract that forces the council to ask for more information.
  • Starting deep GitHub work before the request is aligned with the real workshop idea.
  • Leaving tags, stakeholder ownership, or later publish requirements until the end of the flow.
This step exists so the rest of the work starts from an approved request instead of assumptions.

Step 2 / GitHub build

Create Workshop + Add to GitHub

Set up the workstation, fork and clone the right repositories, copy the sample structure, and make preview work early so the workshop is cheap to fix.

Output: live preview Time: 15 to 20 minutes Leads to: content authoring
Most new-author friction lives here. Get account setup, repository hygiene, folder structure, and preview working before you write long sections of content.

  1. Create or confirm the GitHub account that uses your Oracle email, then finish profile name, username, profile photo, and two-factor authentication in GitHub Settings.
  2. Install GitHub Desktop, open File > Options > Sign in, and confirm the client uses the same account that will fork and clone the repository.

Fork, clone, sync

Build the repository in the same order the canonical guide expects

Keep production, your fork, and your local clone clearly separated. New authoring gets much harder when those roles blur.

  1. After the workshop is approved, fork the target oracle-livelabs product repository and also fork common so you have both the production content repository and the shared sample assets.
  2. Clone your fork with GitHub Desktop and select To contribute to the parent project when prompted so upstream/main stays connected.
  3. Before editing each day, click Fetch origin, then open Branch > Merge into Current Branch and merge upstream/main into main.
  4. After the merge succeeds, click Push origin so your local clone and your fork stay aligned before you author anything new.
  5. Copy the sample workshop structure from common/sample-livelabs-templates/sample-workshop, keep every file and folder lowercase, and point manifest.json at real markdown files instead of placeholders.
Preview early

Enable GitHub Pages and verify the path before deep writing starts

Preview is not a late polish step. It is how you prove the workshop structure, manifest paths, and shared assets already render correctly.

  1. Open your fork on GitHub, go to Settings > Pages, choose the main branch, and save the configuration.
  2. Wait for GitHub Pages to publish, then open the exact workshop variant path you plan to share.
  3. Use local preview while you author, but always confirm that the same workshop path also works on your personal github.io site.
Commands and baseline

Group GitHub commands by the job they perform

Copy only the command group you need. Keep setup, synchronization, and manifest baseline work separate so review problems are easier to diagnose.

Git settings

Configure paths and filename case

Run this once on a workstation before you work with long paths or case-sensitive filenames.

git config --global core.longpaths true
git config --global core.ignorecase false
Repository synchronization

Sync your fork before editing

Use this group before a new authoring session so your fork and local clone start from the current upstream branch.

git remote -v
git fetch upstream
git merge upstream/main -m "Sync with main"
git push origin main

Before you move on

Step 2 exit criteria

  • GitHub Desktop, Visual Studio Code, and Live Server are installed and working with the Oracle-linked GitHub account.
  • The target repository and common are both forked or available, and the local clone still points to upstream/main.
  • Folder names, file names, and manifest paths are lowercase and case-correct.
  • Your personal GitHub Pages URL renders the same workshop path you intend to share for review.
Watch for

Common build-stage mistakes

  • Using the wrong GitHub account or cloning the wrong repository before the workshop path is confirmed.
  • Skipping the daily upstream merge and only discovering conflicts or stale templates at pull request time.
  • Leaving sample manifest fields, mixed-case filenames, or broken tutorial paths behind.
This step combines workstation setup, repository hygiene, sample structure, and preview into one stable build checkpoint.
3
Step 3 / Quality Assurance and release

Review & Publish

Complete Self Quality Assurance in the right order, fix pull request blockers by class, and submit the final publishing details only after the workshop is genuinely ready to move to production.

Output: reviewed release Time: 10 to 15 minutes Leads to: production
The last step is about precision, not volume. Reviewers should see one stable preview, one saved checklist, one clean pull request, and the exact production URLs they expect.
Self Quality Assurance order

Run Self Quality Assurance in WMS before you open the final publish handoff

This is the part that most often breaks the process. Save the checklist first, then certify the status change. Publishing only makes sense after Self Quality Assurance is complete and stakeholder review is underway or finished.

  1. Open the personal github.io workshop URL, append ?qa=true, and run the workshop end to end before you ask anyone else to review it.
  2. On the WMS details page, update title, short description, long description, outline, prerequisites, tags, and Development GitHub/GitLab URL so WMS matches the actual workshop reviewers will open.
  3. Open Self Quality Assurance Checklist, check every required box, upload the requested evidence, add the pull request link when you create one, and add your personal github.io workshop link.
  4. Click Save on the checklist first. Only after the save succeeds should you move to Self Quality Assurance Complete and certify the change.
  5. Stakeholders verify the workshop from Self Quality Assurance Complete. Once they move it to Completed, the workshop is in the right place for final publishing handling.
Status gate

Keep the release flow aligned with the real review state

Publishing is not the next step after draft content. It happens after Self Quality Assurance, stakeholder verification, and a clean pull request handoff.

Self Quality Assurance

The workshop is stable enough to test end to end and complete the checklist.

Self Quality Assurance Complete

The checklist is saved and certified, and stakeholders receive the workshop for verification.

Completed

Stakeholders have approved the handoff and the workshop is ready for publishing action.

Publish Requested / Approved

The production publishing entry is in motion and the merged workshop can roll out.

Quarterly Quality Assurance

Published workshops cycle back into Quality Assurance later, so keep WMS accurate after release too.

Pull request checks

Fix review blockers by class instead of guessing

When the pull request shows a red X, name the failing workflow first. Image-size problems, markdown-structure problems, and publishing metadata problems are different repairs.

  1. If the pull request fails LiveLabs Image Validation, resize the blocking image to 1280px or less in either dimension and rerun the checks.
  2. If the pull request fails LiveLabs Markdown Validation, fix the exact file and rule: required sections, task format, alt text, filename case, copy tags, or inline HTML.
  3. Use the local validator when you want faster repair loops than waiting for GitHub Actions after every commit.
Validator commands

Run the markdown validator locally before the next push

Choose the command block for your terminal environment and copy only that block.

Bash

Linux or macOS terminal

Use this command block when Bash is your normal terminal.

curl -O https://raw.githubusercontent.com/oracle-livelabs/common/main/md-validator/.github/scripts/validate-livelabs-markdown.sh
chmod +x validate-livelabs-markdown.sh
./validate-livelabs-markdown.sh /path/to/workshop
PowerShell

Windows terminal

Use this command block on Windows and keep the execution policy bypass scoped to the current process.

Invoke-WebRequest -Uri "https://raw.githubusercontent.com/oracle-livelabs/common/main/md-validator/.github/scripts/validate-livelabs-markdown.ps1" -OutFile "validate-livelabs-markdown.ps1"
Set-ExecutionPolicy -ExecutionPolicy Bypass -Scope Process
.\validate-livelabs-markdown.ps1 C:\path\to\your\workshop
Publish handoff

Open the pull request with the WMS ID and fill the Publishing tab with production URLs

The preview URL belongs in development review. The oracle-livelabs URL belongs in production publishing. Keep those two surfaces separate.

  1. Create the pull request from GitHub Desktop after the Quality Assurance fixes are pushed, and include the WMS ID in the pull request title because review will not start without it.
  2. On the pull request page, fill the required checklist and general requirements from the Quality Assurance handoff so reviewers can tell the workshop is ready for merge.
  3. Confirm the branch is up to date with main and has no merge conflicts before you request approval.
  4. Open WMS > Publishing, click Publish to LiveLabs, set the publish type and workshop time, and replace any personal preview URL with the final oracle-livelabs production path.
  5. After merge and publishing approval, verify the production workshop and leave WMS aligned for the next Quarterly Quality Assurance cycle.

Before you move on

Step 3 exit criteria

  • The preview works end to end with ?qa=true and the WMS metadata matches the real workshop.
  • The Self Quality Assurance checklist is saved before any status change and stakeholder verification has a clean handoff surface.
  • The pull request title includes the WMS ID, the branch is current, and the failing check class is understood if continuous integration is red.
  • The Publishing tab uses oracle-livelabs production URLs, not the personal preview link.
  • The production workshop is verified after merge, and WMS stays accurate for future Quarterly Quality Assurance.
Watch for

Common release-stage mistakes

  • Sharing a preview before checking tutorial order, links, images, and copy flows end to end.
  • Changing WMS status before the checklist is saved and certified.
  • Mixing development and production URLs or opening a pull request from a stale branch.
At the end of this step, the workshop is in a clean state for review, publishing, and later Quarterly Quality Assurance follow-up.
