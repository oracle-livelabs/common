# Oracle Row-Level Security (Virtual Private Database)

## Overview

Virtual Private Database (VPD), also called Fine-Grained Access Control (FGAC), is Oracle's mechanism for enforcing row-level security transparently at the database kernel level. Unlike application-layer filtering — which can be bypassed by ad-hoc queries, ETL tools, or reporting applications — VPD policies are enforced by the Oracle query engine itself, regardless of how a query reaches the database.

When a VPD policy is applied to a table, Oracle automatically rewrites every SQL statement that touches that table to include a `WHERE` clause generated by a PL/SQL policy function. This happens before the optimizer even sees the query. The result is that a user literally cannot read, update, or delete rows they are not authorized to access.

VPD is a licensed feature included with **Oracle Database Enterprise Edition**. It is used extensively in multi-tenant SaaS applications, healthcare systems (HIPAA), and financial systems (SOX, PCI-DSS).

---

## Architecture Overview

```
Application sends:  SELECT * FROM hr.employees

Oracle VPD intercepts and rewrites to:
  SELECT * FROM hr.employees
  WHERE department_id IN (
    SELECT department_id FROM hr.user_dept_access
    WHERE username = SYS_CONTEXT('USERENV', 'SESSION_USER')
  )

The rewritten query is what actually executes.
```

The components of a VPD implementation are:

1. **Application Context** — A named set of session-level attributes (key-value pairs) used to hold the security context for the current user (e.g., tenant ID, user role, department).
2. **Policy Function** — A PL/SQL function that returns a `VARCHAR2` predicate string. This is the `WHERE` clause fragment that Oracle will append.
3. **Policy** — Registered via `DBMS_RLS.ADD_POLICY`, associates a table/view with a policy function and configures when it fires.

## Application Contexts

Application contexts are the primary mechanism for storing session-specific security attributes that policy functions read. They are namespaced and can only be set by a designated trusted package.

### Creating an Application Context

```sql
-- Create the context namespace
-- The USING clause names the package authorized to set values in this context
CREATE OR REPLACE CONTEXT hr_security_ctx USING hr.set_context_pkg;

-- Separate context used to drive policy groups
CREATE OR REPLACE CONTEXT hr_app_ctx USING hr.set_context_pkg ACCESSED GLOBALLY;
```

### Creating the Context-Setting Package

```sql
CREATE OR REPLACE PACKAGE hr.set_context_pkg AS
  PROCEDURE set_user_context;
  PROCEDURE set_policy_group(p_group IN VARCHAR2);
END set_context_pkg;
/

CREATE OR REPLACE PACKAGE BODY hr.set_context_pkg AS

  PROCEDURE set_user_context IS
    v_dept_id    NUMBER;
    v_user_role  VARCHAR2(30);
    v_tenant_id  NUMBER;
  BEGIN
    -- Look up the current user's department and role
    SELECT e.department_id, e.job_id
    INTO v_dept_id, v_user_role
    FROM hr.employees e
    WHERE e.email = SYS_CONTEXT('USERENV', 'SESSION_USER');

    -- Set the context values for this session
    DBMS_SESSION.SET_CONTEXT('hr_security_ctx', 'dept_id',   v_dept_id);
    DBMS_SESSION.SET_CONTEXT('hr_security_ctx', 'user_role', v_user_role);

  EXCEPTION
    WHEN NO_DATA_FOUND THEN
      -- Non-employee users get no access — set dept to -1
      DBMS_SESSION.SET_CONTEXT('hr_security_ctx', 'dept_id',   '-1');
      DBMS_SESSION.SET_CONTEXT('hr_security_ctx', 'user_role', 'NONE');
  END set_user_context;

  PROCEDURE set_policy_group(p_group IN VARCHAR2) IS
  BEGIN
    DBMS_SESSION.SET_CONTEXT('hr_app_ctx', 'policy_group', p_group);
  END set_policy_group;

END set_context_pkg;
/
```

### Setting Context Values via a Logon Trigger

The most reliable way to populate the context is via a logon trigger, which fires automatically when any session is established:

```sql
CREATE OR REPLACE TRIGGER hr.set_logon_context
  AFTER LOGON ON DATABASE
BEGIN
  -- Only set context for non-DBA sessions to the HR application
  IF SYS_CONTEXT('USERENV', 'SESSION_USER') NOT IN ('SYS', 'SYSTEM') THEN
    hr.set_context_pkg.set_user_context;
  END IF;
EXCEPTION
  WHEN OTHERS THEN
    NULL;  -- Do not prevent login on context errors; log instead
END;
/
```

### Reading Context Values

```sql
-- In policy functions and SQL queries
SYS_CONTEXT('hr_security_ctx', 'dept_id')
SYS_CONTEXT('hr_security_ctx', 'user_role')

-- Built-in USERENV namespace values (always available)
SYS_CONTEXT('USERENV', 'SESSION_USER')    -- Database username
SYS_CONTEXT('USERENV', 'OS_USER')         -- OS username
SYS_CONTEXT('USERENV', 'IP_ADDRESS')      -- Client IP
SYS_CONTEXT('USERENV', 'MODULE')          -- Application module name
SYS_CONTEXT('USERENV', 'CLIENT_INFO')     -- Application client info
SYS_CONTEXT('USERENV', 'HOST')            -- Client hostname
SYS_CONTEXT('USERENV', 'DB_NAME')         -- Database name
SYS_CONTEXT('USERENV', 'CURRENT_SCHEMA')  -- Current schema
SYS_CONTEXT('USERENV', 'ISDBA')           -- 'TRUE' if logged in as DBA
```

## Policy Functions

A policy function must have a specific signature: it accepts the schema name and object name as parameters and returns a `VARCHAR2` predicate string. If it returns `NULL` or an empty string, no restriction is applied (all rows visible). If it returns `'1=0'`, no rows are visible.

### Simple Department-Based Policy

```sql
CREATE OR REPLACE FUNCTION hr.emp_dept_policy(
  p_schema  IN VARCHAR2,
  p_object  IN VARCHAR2
) RETURN VARCHAR2 AS
  v_predicate VARCHAR2(4000);
  v_dept_id   VARCHAR2(10);
  v_role      VARCHAR2(30);
BEGIN
  v_dept_id := SYS_CONTEXT('hr_security_ctx', 'dept_id');
  v_role    := SYS_CONTEXT('hr_security_ctx', 'user_role');

  -- HR Managers see their department
  -- HR Directors see all departments
  -- Everyone else sees only their own record
  IF v_role = 'HR_MGR' THEN
    v_predicate := 'department_id = ' || v_dept_id;
  ELSIF v_role = 'HR_DIR' THEN
    v_predicate := NULL;  -- No restriction; all rows visible
  ELSE
    -- Regular employees see only their own row
    v_predicate := 'email = SYS_CONTEXT(''USERENV'', ''SESSION_USER'')';
  END IF;

  RETURN v_predicate;
END emp_dept_policy;
/
```

### Multi-Tenant Partition Policy

```sql
CREATE OR REPLACE FUNCTION saas.tenant_isolation_policy(
  p_schema  IN VARCHAR2,
  p_object  IN VARCHAR2
) RETURN VARCHAR2 AS
BEGIN
  -- If context not set (DBA bypass or ETL), return impossible predicate
  IF SYS_CONTEXT('saas_ctx', 'tenant_id') IS NULL THEN
    -- Allow DBAs unrestricted access
    IF SYS_CONTEXT('USERENV', 'ISDBA') = 'TRUE' THEN
      RETURN NULL;
    END IF;
    RETURN '1=0';  -- No context = no rows
  END IF;

  RETURN 'tenant_id = ' || SYS_CONTEXT('saas_ctx', 'tenant_id');
END tenant_isolation_policy;
/
```

## Registering Policies with DBMS_RLS

### Basic Policy Registration

```sql
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'HR',
    object_name     => 'EMPLOYEES',
    policy_name     => 'EMP_DEPT_VISIBILITY',
    function_schema => 'HR',
    policy_function => 'EMP_DEPT_POLICY',
    statement_types => 'SELECT, INSERT, UPDATE, DELETE',
    update_check    => TRUE,   -- Prevent UPDATE from moving rows out of scope
    enable          => TRUE
  );
END;
/
```

### Policy Parameters Explained

| Parameter | Description |
|---|---|
| `object_schema` | Schema of the protected table/view |
| `object_name` | Name of the protected table, view, or synonym |
| `policy_name` | Unique name for this policy |
| `function_schema` | Schema where the policy function lives |
| `policy_function` | Name of the policy function |
| `statement_types` | Comma-separated: `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `INDEX` |
| `update_check` | When TRUE, prevents INSERT/UPDATE from creating rows outside the policy predicate |
| `enable` | Whether policy is active immediately |
| `static_policy` | When TRUE, function called once and result cached — highest performance |
| `policy_type` | Overrides `static_policy` with a richer type constant |
| `long_predicate` | When TRUE, allows predicates up to 32KB |
| `sec_relevant_cols` | Comma-separated columns; policy only fires if these are referenced |
| `sec_relevant_cols_opt` | When `DBMS_RLS.ALL_ROWS`, returns rows with NULL in restricted columns instead of hiding them |

## Policy Types

Oracle provides several policy types that control when the policy function is called and whether its result is cached. Choosing the right type is critical for performance.

```sql
-- Policy type constants in DBMS_RLS:
-- DBMS_RLS.STATIC             -- Function called once; result cached for lifetime of policy
-- DBMS_RLS.SHARED_STATIC      -- Single cached predicate shared across all users
-- DBMS_RLS.CONTEXT_SENSITIVE  -- Re-evaluated when application context changes
-- DBMS_RLS.SHARED_CONTEXT_SENSITIVE -- Shared predicate, re-evaluated on context change
-- DBMS_RLS.DYNAMIC            -- Called fresh for every query (default)

-- Static policy: highest performance, predicate never changes per user
-- Example: a global filter like 'status != ''DELETED'''
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'APP',
    object_name     => 'ORDERS',
    policy_name     => 'HIDE_DELETED_ORDERS',
    function_schema => 'APP',
    policy_function => 'ACTIVE_RECORDS_POLICY',
    statement_types => 'SELECT',
    policy_type     => DBMS_RLS.STATIC
  );
END;
/

-- Context-sensitive policy: re-evaluated when the application context changes
-- Best for app-context-based multi-tenancy
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'SAAS',
    object_name     => 'CUSTOMER_DATA',
    policy_name     => 'TENANT_ISOLATION',
    function_schema => 'SAAS',
    policy_function => 'TENANT_ISOLATION_POLICY',
    statement_types => 'SELECT, INSERT, UPDATE, DELETE',
    update_check    => TRUE,
    policy_type     => DBMS_RLS.CONTEXT_SENSITIVE
  );
END;
/

-- Dynamic policy: called for every query
-- Use when the predicate depends on data looked up at query time
-- More flexible but slower — avoid for high-frequency tables
BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema   => 'HR',
    object_name     => 'SALARY_HISTORY',
    policy_name     => 'SALARY_ACCESS',
    function_schema => 'HR',
    policy_function => 'SALARY_POLICY',
    statement_types => 'SELECT',
    policy_type     => DBMS_RLS.DYNAMIC
  );
END;
/
```

## Column-Masking Policies (sec_relevant_cols_opt)

Instead of hiding entire rows, you can return rows with NULL in sensitive columns for unauthorized users:

```sql
CREATE OR REPLACE FUNCTION hr.salary_col_policy(
  p_schema  IN VARCHAR2,
  p_object  IN VARCHAR2
) RETURN VARCHAR2 AS
BEGIN
  -- HR and payroll roles see salaries; others see NULLs in salary column
  IF SYS_CONTEXT('hr_security_ctx', 'user_role') IN ('HR_MGR', 'PAYROLL') THEN
    RETURN NULL;  -- No restriction
  END IF;
  RETURN '1=0';  -- Will be overridden by sec_relevant_cols_opt = ALL_ROWS
END salary_col_policy;
/

BEGIN
  DBMS_RLS.ADD_POLICY(
    object_schema          => 'HR',
    object_name            => 'EMPLOYEES',
    policy_name            => 'MASK_SALARY',
    function_schema        => 'HR',
    policy_function        => 'SALARY_COL_POLICY',
    statement_types        => 'SELECT',
    sec_relevant_cols      => 'SALARY,COMMISSION_PCT',
    sec_relevant_cols_opt  => DBMS_RLS.ALL_ROWS  -- Return rows with NULL salary
  );
END;
/

-- Employees outside the HR/Payroll roles will see:
-- EMPLOYEE_ID | LAST_NAME | SALARY | COMMISSION_PCT
-- 100         | King      | (null) | (null)
-- 101         | Kochhar   | (null) | (null)
```

## Managing Policies

```sql
-- Enable a disabled policy
EXEC DBMS_RLS.ENABLE_POLICY('HR', 'EMPLOYEES', 'EMP_DEPT_VISIBILITY', TRUE);

-- Disable a policy (DBA bypass for maintenance)
EXEC DBMS_RLS.ENABLE_POLICY('HR', 'EMPLOYEES', 'EMP_DEPT_VISIBILITY', FALSE);

-- Drop a policy
EXEC DBMS_RLS.DROP_POLICY('HR', 'EMPLOYEES', 'EMP_DEPT_VISIBILITY');

-- Refresh a static or context-sensitive policy (force re-evaluation of cached predicate)
EXEC DBMS_RLS.REFRESH_POLICY('HR', 'EMPLOYEES', 'EMP_DEPT_VISIBILITY');

-- Refresh all policies on a table
EXEC DBMS_RLS.REFRESH_POLICY('HR', 'EMPLOYEES');

-- List all policies on a table
SELECT object_owner, object_name, policy_name, function, sel, ins, upd, del,
       enable, static_policy, policy_type, long_predicate
FROM dba_policies
WHERE object_owner = 'HR'
  AND object_name  = 'EMPLOYEES'
ORDER BY policy_name;

-- List all policies in the database
SELECT object_owner, object_name, policy_name, function, enable, policy_type
FROM dba_policies
ORDER BY object_owner, object_name;
```

## Groups of Policies

Multiple policies on the same object are combined with AND. You can also use policy groups for mutually exclusive policies (only one group active at a time):

```sql
-- Create a named policy group
EXEC DBMS_RLS.CREATE_POLICY_GROUP('HR', 'EMPLOYEES', 'INTERNAL_GROUP');
EXEC DBMS_RLS.CREATE_POLICY_GROUP('HR', 'EMPLOYEES', 'EXTERNAL_GROUP');

-- Add policies to each group
BEGIN
  DBMS_RLS.ADD_GROUPED_POLICY(
    object_schema   => 'HR',
    object_name     => 'EMPLOYEES',
    policy_group    => 'INTERNAL_GROUP',
    policy_name     => 'INTERNAL_FILTER',
    function_schema => 'HR',
    policy_function => 'INTERNAL_EMP_POLICY'
  );
END;
/

-- Associate a driving application context with the policy groups on this object.
-- Oracle reads the context value at runtime to determine the active policy group.
-- Use DBMS_RLS.ADD_POLICY_CONTEXT to register which context attribute drives group selection:
EXEC DBMS_RLS.ADD_POLICY_CONTEXT('HR', 'EMPLOYEES', 'HR_APP_CTX', 'POLICY_GROUP');

-- Then, at session time, call the trusted package to set the active group:
EXEC hr.set_context_pkg.set_policy_group('INTERNAL_GROUP');

-- There is no DBMS_RLS.SET_CONTEXT procedure. Policy group activation is driven by
-- an application context value registered via ADD_POLICY_CONTEXT.
```

## Best Practices

1. **Always set context via a logon trigger**: Relying on applications to call `DBMS_SESSION.SET_CONTEXT` before queries opens a window where queries without context return no rows (or all rows, depending on your policy). A logon trigger eliminates this gap.

2. **Use `CONTEXT_SENSITIVE` or `STATIC` policy types**: `DYNAMIC` policies are re-evaluated on every query and can become a significant performance bottleneck on busy tables. Use them only when the predicate must change within a session.

3. **Return `'1=0'` for missing context, not NULL**: If the context value is not set, returning NULL removes the policy predicate and exposes all rows. Returning `'1=0'` ensures the default is deny-all.

4. **Set `update_check => TRUE`**: This prevents users from INSERTing or UPDATEing rows outside their visible window, which would create orphaned rows they can never access or modify again.

5. **Never put DML in a policy function**: Policy functions are called by the Oracle kernel during query parsing. DML in a policy function causes recursive SQL and errors.

6. **Use views for simpler cases**: If your filtering logic is static or based on simple joins, a view with a `WHERE` clause may be cleaner and faster than VPD.

7. **Document every policy**: Include the business rule being enforced as a comment in the policy function. VPD is invisible to query writers and must be well-documented.

## Common Mistakes and How to Avoid Them

### Mistake 1: Returning NULL When Context Is Missing

```sql
-- BAD: Returns NULL when context not set, removing the filter
CREATE OR REPLACE FUNCTION bad_policy(p_schema VARCHAR2, p_object VARCHAR2)
RETURN VARCHAR2 AS
BEGIN
  RETURN 'tenant_id = ' || SYS_CONTEXT('app_ctx', 'tenant_id');
  -- If tenant_id is NULL, this returns 'tenant_id = ' which is a SQL error
  -- and if the policy function raises an exception, Oracle raises an error for the statement
END;
/

-- GOOD: Explicit NULL check with fail-safe
CREATE OR REPLACE FUNCTION good_policy(p_schema VARCHAR2, p_object VARCHAR2)
RETURN VARCHAR2 AS
  v_tenant VARCHAR2(10) := SYS_CONTEXT('app_ctx', 'tenant_id');
BEGIN
  IF v_tenant IS NULL THEN
    RETURN '1=0';  -- Default deny
  END IF;
  RETURN 'tenant_id = ' || TO_NUMBER(v_tenant);  -- Cast to prevent injection
END;
/
```

### Mistake 2: SQL Injection in Policy Predicates

Policy functions build dynamic SQL predicates. If you concatenate user-supplied values, you may introduce SQL injection into the WHERE clause:

```sql
-- POTENTIALLY DANGEROUS: Trust only values from SYS_CONTEXT (not user input)
-- SYS_CONTEXT values can only be set by the trusted package, making them safe
RETURN 'dept_id = ' || SYS_CONTEXT('app_ctx', 'dept_id');  -- Safe: context is trusted

-- DANGEROUS: Never build predicates from raw user input
RETURN 'dept_id = ' || v_user_supplied_param;  -- Never do this
```

### Mistake 3: Performance Impact from DYNAMIC Policies

```sql
-- Profile VPD policy performance
SELECT sql_text, executions, elapsed_time/1000000 elapsed_sec,
       rows_processed
FROM v$sql
WHERE sql_text LIKE '%EMPLOYEES%'
  AND parsing_user_id != 0
ORDER BY elapsed_time DESC;

-- If policy function is slow, check for missing indexes on the context lookup tables
-- Use CONTEXT_SENSITIVE with a logon trigger instead of DYNAMIC
```

### Mistake 4: Forgetting Policies Apply to Views and Synonyms

VPD policies are attached to the base table. A view or synonym over that table inherits the policy automatically. However, policies can also be added directly to views, which can cause double-filtering if not designed carefully.

## Compliance Considerations

### HIPAA
VPD is an ideal control for the HIPAA "minimum necessary" standard. Patient records can be restricted so clinicians see only their assigned patients, billing sees only billing-relevant fields, and administrative staff see only demographics.

### PCI-DSS
PCI Requirement 7 (restrict access to cardholder data by business need to know) is directly addressable with VPD. Cardholder data tables can have a policy that restricts visibility to only the payment processing application's session context.

### SOX
Multi-entity financial databases can use VPD to enforce that users in one business unit cannot see the financial records of another, satisfying separation-of-duties requirements at the data layer rather than the application layer.

```sql
-- Example: SOX-compliant entity-level isolation
CREATE OR REPLACE FUNCTION finance.entity_isolation_policy(
  p_schema  IN VARCHAR2,
  p_object  IN VARCHAR2
) RETURN VARCHAR2 AS
  v_entity_id VARCHAR2(10);
  v_is_auditor VARCHAR2(5);
BEGIN
  v_entity_id  := SYS_CONTEXT('finance_ctx', 'entity_id');
  v_is_auditor := SYS_CONTEXT('finance_ctx', 'is_auditor');

  -- External auditors need cross-entity visibility
  IF v_is_auditor = 'TRUE' THEN
    RETURN NULL;
  END IF;

  IF v_entity_id IS NULL THEN
    RETURN '1=0';
  END IF;

  RETURN 'legal_entity_id = ' || TO_NUMBER(v_entity_id);
END entity_isolation_policy;
/
```
