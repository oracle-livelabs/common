# Sequences and Identity Columns in Oracle Database

## Overview

Generating unique, monotonically increasing identifiers is a fundamental requirement in relational database design. Oracle provides two mechanisms:

- **Sequences** — standalone schema objects that generate numbers independently of any table
- **Identity Columns** — column-level syntax that combines sequence generation with table DDL (introduced in Oracle 12c)

Both ultimately use the same internal sequence engine, but identity columns provide a cleaner, more declarative way to define surrogate keys.

---

## CREATE SEQUENCE: Full Syntax and Options

```sql
CREATE SEQUENCE schema.sequence_name
    START WITH      initial_value        -- first value generated (default: 1)
    INCREMENT BY    step_value           -- positive = ascending, negative = descending (default: 1)
    MINVALUE        min_value            -- lower bound
    MAXVALUE        max_value            -- upper bound (default: 10^27 for ASC)
    NOCYCLE | CYCLE                      -- NOCYCLE raises ORA-08004 at limit (default: NOCYCLE)
    CACHE n | NOCACHE                    -- pre-allocate n values in memory (default: 20)
    ORDER | NOORDER                      -- ORDER guarantees generation order (only matters for RAC)
    KEEP | NOKEEP                        -- affects sequence behavior after session restore
    SCALE | NOSCALE;                     -- introduced in 23ai: prepend instance/shard info for uniqueness
```

### Basic Examples

```sql
-- Simple primary key sequence (most common pattern)
CREATE SEQUENCE seq_customer_id
    START WITH 1000
    INCREMENT BY 1
    MAXVALUE 9999999999
    NOCYCLE
    CACHE 100;

-- Descending sequence
CREATE SEQUENCE seq_priority_desc
    START WITH 10000
    INCREMENT BY -1
    MINVALUE 1
    NOCYCLE
    CACHE 50;

-- Step sequence for generating even numbers
CREATE SEQUENCE seq_even_numbers
    START WITH 2
    INCREMENT BY 2
    MAXVALUE 1000000
    NOCYCLE
    CACHE 20;
```

### Using a Sequence

```sql
-- NEXTVAL: advances the sequence and returns the new value
-- CURRVAL: returns the value last generated by NEXTVAL in this session (no advance)

-- In an INSERT
INSERT INTO customers (customer_id, name, email)
VALUES (seq_customer_id.NEXTVAL, 'Acme Corp', 'admin@acme.com');

-- Capture the generated value in PL/SQL
DECLARE
    v_new_id customers.customer_id%TYPE;
BEGIN
    INSERT INTO customers (customer_id, name, email)
    VALUES (seq_customer_id.NEXTVAL, 'Beta LLC', 'info@beta.com')
    RETURNING customer_id INTO v_new_id;

    -- Use v_new_id for child record inserts
    INSERT INTO customer_contacts (contact_id, customer_id, contact_name)
    VALUES (seq_contacts.NEXTVAL, v_new_id, 'John Smith');

    COMMIT;
END;

-- In a SELECT (useful for generating row numbers without a table)
SELECT seq_customer_id.NEXTVAL FROM DUAL;

-- CURRVAL in the same session after NEXTVAL has been called
SELECT seq_customer_id.CURRVAL FROM DUAL;
```

### Altering and Dropping Sequences

```sql
-- Change cache size
ALTER SEQUENCE seq_customer_id CACHE 500;

-- Restart at a new value (useful after data migration)
ALTER SEQUENCE seq_customer_id RESTART START WITH 50000;  -- 18c+

-- Pre-18c: approximate restart by setting INCREMENT to a large value
SELECT last_number FROM user_sequences WHERE sequence_name = 'SEQ_CUSTOMER_ID';
-- If current is 1000 and you want to restart at 50000, set increment to 49000
ALTER SEQUENCE seq_customer_id INCREMENT BY 49000;
SELECT seq_customer_id.NEXTVAL FROM DUAL;  -- now at 50000
ALTER SEQUENCE seq_customer_id INCREMENT BY 1;

-- Drop a sequence
DROP SEQUENCE seq_customer_id;
```

## The CACHE Option: Performance Impact

`CACHE` is the most impactful sequence parameter for performance. Oracle pre-allocates `CACHE` values in the SGA. Incrementing a cached sequence is a pure in-memory operation — extremely fast. Without caching (`NOCACHE`), every `NEXTVAL` requires a write to the `seq$` system table.

### Performance Comparison

| Configuration | NEXTVAL Cost | Notes |
|---|---|---|
| `NOCACHE` | ~1 redo log write | ~100x slower; avoid in OLTP |
| `CACHE 20` (default) | In-memory increment | Good for low-volume |
| `CACHE 100` | In-memory increment | Good for moderate OLTP |
| `CACHE 1000+` | In-memory increment | Bulk inserts, high-volume |

```sql
-- Check sequence cache performance in AWR
SELECT object_name, value AS total_waits
FROM   v$sesstat st
JOIN   v$statname sn ON st.statistic# = sn.statistic#
WHERE  sn.name = 'sequence misses'
  AND  st.sid = SYS_CONTEXT('USERENV', 'SID');
```

### Cache Loss on Instance Restart

When the database restarts, cached but unused sequence values are **discarded**. If you cached 1000 values and used 50, the next session starts at value 1051 — a gap of 950. This is normal and expected. If your application cannot tolerate gaps, you cannot use CACHE (and will pay the performance penalty).

```sql
-- How many values will be lost on restart?
SELECT sequence_name, last_number, cache_size,
       last_number + cache_size - 1 AS max_cached_value,
       (cache_size - 1) AS max_possible_gap
FROM   user_sequences
WHERE  sequence_name = 'SEQ_CUSTOMER_ID';
```

## Gaps in Sequences

Sequence values are **not gap-free** by design. Gaps occur because:

1. **Cache loss on restart** — unused cached values are discarded
2. **Rollback** — `NEXTVAL` is never rolled back; a rolled-back INSERT wastes the value
3. **CYCLE** — when the sequence wraps around
4. **DELETE** — rows (and their sequence values) can be deleted

```sql
-- Sequence values are NOT rolled back
BEGIN
    INSERT INTO orders (order_id) VALUES (seq_orders.NEXTVAL);  -- uses 1001
    ROLLBACK;  -- INSERT undone, but 1001 is gone
END;

BEGIN
    INSERT INTO orders (order_id) VALUES (seq_orders.NEXTVAL);  -- uses 1002
    COMMIT;
END;
-- Gap: 1001 is missing from orders table
```

**If gap-free numbering is required:** Use a different mechanism, such as a dedicated table with a row lock or `DBMS_LOCK` for serialization. This is almost always a mistake — gap-free sequences require serial execution and are a performance bottleneck. Business requirements for "no gaps" in numbering (e.g., invoice numbers for tax compliance) should use a separate post-commit numbering step.

## ORDER vs. NOORDER (RAC Environments)

In Oracle RAC (Real Application Clusters), multiple instances can generate sequence values independently from their own cache. With `NOORDER` (default), values are generated in no guaranteed order across nodes — a node 1 might generate 100–199, node 2 generates 200–299, and requests could get 150, 250, 160, 210 in interleaved order.

With `ORDER`, Oracle ensures strictly ordered generation across all RAC nodes by serializing through the cluster interconnect. This has significant performance implications.

```sql
-- ORDER sequence (only needed for RAC when strict ordering matters)
CREATE SEQUENCE seq_ordered_id
    ORDER
    CACHE 10;  -- smaller cache beneficial with ORDER; large cache wastes values

-- For most applications on RAC, NOORDER + CACHE is the right choice
CREATE SEQUENCE seq_fast_id
    NOORDER
    CACHE 500;
```

## Identity Columns (12c+)

Identity columns are the modern way to define auto-incrementing primary keys. Oracle automatically creates and manages an underlying sequence.

### GENERATED ALWAYS

The column value is **always** generated by Oracle. You cannot insert or update the column directly.

```sql
CREATE TABLE employees (
    employee_id   NUMBER GENERATED ALWAYS AS IDENTITY
                      (START WITH 1000 INCREMENT BY 1 CACHE 100),
    first_name    VARCHAR2(50) NOT NULL,
    last_name     VARCHAR2(50) NOT NULL,
    hire_date     DATE DEFAULT SYSDATE NOT NULL,
    CONSTRAINT pk_employees PRIMARY KEY (employee_id)
);

-- Correct INSERT: omit the identity column
INSERT INTO employees (first_name, last_name) VALUES ('Jane', 'Doe');

-- Wrong: explicit value rejected
INSERT INTO employees (employee_id, first_name, last_name)
VALUES (9999, 'Jane', 'Doe');
-- ORA-32795: cannot insert into a generated always identity column
```

### GENERATED BY DEFAULT

Oracle generates the value if you omit the column. You can override with an explicit value.

```sql
CREATE TABLE products (
    product_id    NUMBER GENERATED BY DEFAULT AS IDENTITY,
    product_name  VARCHAR2(100) NOT NULL,
    sku           VARCHAR2(50)  NOT NULL UNIQUE
);

-- Oracle generates the ID
INSERT INTO products (product_name, sku) VALUES ('Widget Pro', 'WGT-001');

-- Override with explicit value (useful for data migration)
INSERT INTO products (product_id, product_name, sku) VALUES (9999, 'Legacy Widget', 'WGT-LEG');
```

### GENERATED BY DEFAULT ON NULL

Like `GENERATED BY DEFAULT`, but also generates a value when `NULL` is explicitly inserted:

```sql
CREATE TABLE sessions (
    session_id    NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY,
    session_token VARCHAR2(64) NOT NULL,
    created_at    TIMESTAMP DEFAULT SYSTIMESTAMP
);

-- NULL triggers generation
INSERT INTO sessions (session_id, session_token) VALUES (NULL, 'abc123');
-- Oracle generates session_id instead of inserting NULL
```

### Retrieving Generated Identity Values

```sql
-- In PL/SQL using RETURNING
DECLARE
    v_new_id employees.employee_id%TYPE;
BEGIN
    INSERT INTO employees (first_name, last_name)
    VALUES ('Bob', 'Smith')
    RETURNING employee_id INTO v_new_id;

    DBMS_OUTPUT.PUT_LINE('New employee ID: ' || v_new_id);
    COMMIT;
END;
```

```java
// In JDBC
String sql = "INSERT INTO employees (first_name, last_name) VALUES (?, ?)";
try (PreparedStatement ps = conn.prepareStatement(sql, new String[]{"EMPLOYEE_ID"})) {
    ps.setString(1, "Bob");
    ps.setString(2, "Smith");
    ps.executeUpdate();

    try (ResultSet generatedKeys = ps.getGeneratedKeys()) {
        if (generatedKeys.next()) {
            long newId = generatedKeys.getLong(1);
            System.out.println("Generated ID: " + newId);
        }
    }
}
```

### Viewing the Underlying Sequence

```sql
-- Identity columns have an underlying sequence managed by Oracle
SELECT column_name, identity_column, data_default,
       generation_type, sequence_name, increment_by, start_with, cache_size
FROM   user_tab_identity_cols
WHERE  table_name = 'EMPLOYEES';
```

## SYS_GUID() vs. Sequences for UUIDs

Oracle provides `SYS_GUID()` (System Globally Unique Identifier) as an alternative to sequences for generating unique identifiers. SYS_GUID returns a 16-byte RAW value (displayed as 32 hex characters).

### Comparison

| Feature | Sequence | SYS_GUID() |
|---|---|---|
| Data type | NUMBER | RAW(16) |
| Value size | ~10 bytes | 16 bytes |
| Index efficiency | Excellent (monotonic) | Poor (random, causes index fragmentation) |
| Globally unique | Only per DB | Yes |
| Human readable | Yes | No (hex string) |
| Performance | Excellent (cached) | Good (but random) |
| Cross-database merge | Conflicts possible | Unique |

```sql
-- SYS_GUID() usage
CREATE TABLE distributed_events (
    event_id    RAW(16) DEFAULT SYS_GUID() PRIMARY KEY,
    event_type  VARCHAR2(50),
    payload     CLOB,
    created_at  TIMESTAMP DEFAULT SYSTIMESTAMP
);

INSERT INTO distributed_events (event_type, payload)
VALUES ('ORDER_PLACED', '{"order_id": 1001}');

-- Display as readable UUID
SELECT RAWTOHEX(event_id) AS event_id_hex,
       REGEXP_REPLACE(RAWTOHEX(event_id),
           '([A-F0-9]{8})([A-F0-9]{4})([A-F0-9]{4})([A-F0-9]{4})([A-F0-9]{12})',
           '\1-\2-\3-\4-\5') AS uuid_formatted,
       event_type
FROM   distributed_events;
```

### Standard UUID (VARCHAR2) Approach

If you need RFC 4122 UUID strings (e.g., for REST APIs that expose IDs):

```sql
-- Store as VARCHAR2 with a check constraint
CREATE TABLE api_resources (
    resource_id  VARCHAR2(36) DEFAULT LOWER(
                     REGEXP_REPLACE(RAWTOHEX(SYS_GUID()),
                         '([A-F0-9]{8})([A-F0-9]{4})([A-F0-9]{4})([A-F0-9]{4})([A-F0-9]{12})',
                         '\1-\2-\3-\4-\5')) PRIMARY KEY,
    resource_type VARCHAR2(50),
    data          CLOB
);

-- Or generate in application and pass in
INSERT INTO api_resources (resource_id, resource_type, data)
VALUES (LOWER(REGEXP_REPLACE(RAWTOHEX(SYS_GUID()),
           '([A-F0-9]{8})([A-F0-9]{4})([A-F0-9]{4})([A-F0-9]{4})([A-F0-9]{12})',
           '\1-\2-\3-\4-\5')),
       'ORDER', '{}');
```

### When to Use Each

- **Sequences / Identity columns**: Default choice for primary keys in any application. Fast, compact, index-friendly.
- **SYS_GUID() / UUID**: Use when records will be merged across multiple databases, when IDs are exposed in APIs (avoids enumeration attacks since values are non-sequential), or when generating IDs outside the database before inserting.

**Important**: Random UUIDs as primary keys cause **index fragmentation**. Each new UUID inserts into a random position in the B-tree index, causing frequent page splits. For high-insert tables, this can degrade performance significantly compared to monotonic sequence values. Consider using a sequence for the database key and a UUID for the external-facing API key.

## Best Practices

- **Use `CACHE 100` or higher** for OLTP sequences. The default of 20 is too low for high-insert systems.
- **Prefer identity columns over manual sequence-in-insert** for new tables. The syntax is cleaner and the intent is clearer.
- **Never assume gap-free sequences**. Design your application to tolerate gaps. If business rules require consecutive numbering (invoices, receipts), generate that number in a separate post-commit step.
- **Use `GENERATED ALWAYS`** unless you have a specific reason to override. It prevents accidental manual inserts.
- **For bulk data loads**, set `CACHE` very high (e.g., `CACHE 10000`) before the load and return it to normal afterward to minimize redo contention.
- **In RAC environments**, use `NOORDER` with sufficient CACHE. The performance cost of `ORDER` is rarely worth it.
- **Avoid `NOCYCLE`** unless you have audited the maximum range. Silent wraparound (with `CYCLE`) will corrupt primary key uniqueness.

## Common Mistakes

### Mistake 1: Using NOCACHE for "Auditing" Purposes

Some teams use `NOCACHE` because they want "no gaps" for audit sequences. This is the worst-of-both-worlds approach: it's still not truly gap-free (gaps occur on rollback), and it's vastly slower because every `NEXTVAL` writes to the data dictionary.

### Mistake 2: Using Sequences with CYCLE Without MAXVALUE

```sql
-- DANGEROUS: default MAXVALUE is 10^27; with CYCLE, it eventually wraps
-- to 1, colliding with existing primary key values
CREATE SEQUENCE seq_bad CYCLE;  -- omitted MAXVALUE

-- SAFE: set MAXVALUE to something realistic and handle CYCLE explicitly
-- Or just use NOCYCLE (default) and monitor for exhaustion
```

### Mistake 3: Selecting NEXTVAL in Bulk Without LIMIT

```sql
-- This generates 1 million sequence values instantly — probably not intended
SELECT seq_orders.NEXTVAL FROM some_large_table;
```

### Mistake 4: Relying on CURRVAL Across Sessions

`CURRVAL` returns the last value generated by `NEXTVAL` in **the current session**. It has no meaning in another session or after reconnecting.

### Mistake 5: Forgetting Identity Columns in Data Exports/Imports

When using `GENERATED ALWAYS`, Data Pump exports include the sequence definition. During import, if the target table already has rows, the sequence may generate duplicate values. Always check `LAST_NUMBER` after import and reset the sequence.

```sql
-- After Data Pump import, fix the identity sequence
SELECT max(employee_id) FROM employees;  -- say it's 5432
ALTER TABLE employees MODIFY employee_id GENERATED ALWAYS AS IDENTITY (START WITH LIMIT VALUE);
```
