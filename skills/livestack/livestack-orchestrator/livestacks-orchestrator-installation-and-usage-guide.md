# LiveStacks Orchestrator Installation And Use Guide

## What The Skill Can Do

- turn a full PRD, partial PRD, workbook-style brief, or sparse business prompt into one Oracle-first LiveStacks solution bundle
- synthesize a compact working PRD before detailed build work starts
- plan specialist roles for project management, solution engineering, database design, UI/UX, full-stack delivery, security/platform, documentation, and challenge review
- keep generated applications ORDS-first, Podman-first, Oracle Database 26ai-centered, and aligned to the LiveStacks golden runtime contract
- scaffold and validate the required sibling `guide/` demo runbook deliverable
- enforce production-readiness gates with scaffold-marker checks, semantic validation, A+ grading, package hygiene checks, and release metadata validation
- install bundled helper snapshots for `oracle-db-skills`, `livestack-guide-builder`, and `redwood-creator` when local specialist skills are missing

## Source Of Truth

- `livestacks-orchestrator/` is the installable Codex skill folder.
- `livestacks-orchestrator/SKILL.md` is the orchestration contract Codex reads when the skill triggers.
- `livestacks-orchestrator/README.md` is the public package README.
- `livestacks-orchestrator/SUPPORT.md` documents prerequisites and support boundaries.
- `livestacks-orchestrator/VERSION` is the skill version source.
- `livestacks-orchestrator/CHANGELOG.md` records release changes.
- `livestacks-orchestrator/update.json` points installed copies to the public update manifest.
- `livestacks-orchestrator.update.json` is the wrapper-level public update manifest.
- `ready-to-deploy-archive/livestacks-orchestrator.zip` is the deployable archive artifact.

Keep those surfaces aligned before publishing. Do not update the archive without also validating the expanded skill and refreshing manifest hashes.

## Core Rules

- run the self-update check before substantial orchestration work
- use the embedded `name:` value, `livestacks-orchestrator`, as the installed folder name
- keep the role ledger visible before specialist execution starts
- use delegated subagents only when the user explicitly asks for subagents, delegation, or parallel agent work
- prefer installed specialist skills first, then bundled helper install, then built-in fallback references
- keep `stack/compose.yml` and `stack/.env.example` aligned to the golden runtime baseline unless the user explicitly approves a runtime-contract exception
- do not call a generated bundle production-ready unless validation and grading pass
- do not package transient cache files, macOS metadata, local credentials, generated bytecode, or machine-local config

## Installation Process

Give Codex this prompt:

```text
Install `livestacks-orchestrator` skill into my local Codex skills directory. Inspect `SKILL.md`, use the embedded `name:` `livestacks-orchestrator` as the installed folder name, ignore `__MACOSX`, `__pycache__`, `.pytest_cache`, `*.pyc`, and `*.pyo`, and verify the installed copy after copying.
```

After installation, ask Codex to verify that:

- the installed folder exists as `$CODEX_HOME/skills/livestacks-orchestrator`, or `~/.codex/skills/livestacks-orchestrator` when `CODEX_HOME` is unset
- `SKILL.md` contains `name: livestacks-orchestrator`
- `VERSION` matches the expected release version
- `agents/openai.yaml` exists
- `scripts/check_skill_package.py` passes

## Self-Update Check

From the installed skill directory, run:

```bash
python3 scripts/self_update.py --auto --json
```

If the result reports `"updated": true`, re-read `SKILL.md` from disk before continuing. If the check skips because the manifest, archive, network, or validation path is unavailable, continue with the installed skill only when the requested work is not blocked by that warning.

## How To Prompt It

Start with `$livestacks-orchestrator` and give Codex a PRD, partial PRD, or business brief. For sparse prompts, include at least the industry and pain point.

## What To Include In Your Request

- industry
- pain point
- target users or operators
- desired demo outcome
- source PRD, partial PRD, workbook notes, or implementation notes
- Oracle feature preferences or constraints, if known
- deployment target, such as local Podman, sandbox, tenancy, or customer handoff
- whether replaceable demo data or Use Your Own Data is required
- whether delegated specialist agents are allowed

## Recommended Prompt Patterns

### Build From A Sparse Brief

```text
$livestacks-orchestrator create a production-ready LiveStacks solution. Industry: <industry>. Pain point: <pain-point>. Desired demo outcome: <outcome>. Use local execution unless I explicitly approve delegated agents.
```

### Build From A PRD

```text
$livestacks-orchestrator turn this PRD into a production-ready Oracle-first LiveStacks bundle with stack, database, app, guide, validation evidence, and A+ grading. Use the PRD below as source of truth.
```

### Review A Generated Bundle

```text
$livestacks-orchestrator review this LiveStacks bundle for golden runtime parity, ORDS-first app wiring, Oracle evidence, Use Your Own Data coverage, guide alignment, package hygiene, and A+ grading readiness.
```

## Validation Commands

Run these from `livestacks-orchestrator/` before sharing, zipping, or embedding the skill package:

```bash
python3 scripts/check_skill_package.py
python3 -m unittest discover -s tests -p 'test_*.py'
```

Run these against a generated solution root before calling the generated bundle production-ready:

```bash
python3 scripts/find_scaffold_markers.py <solution-root>
python3 scripts/validate_livestack_bundle.py <solution-root>
python3 scripts/grade_livestack_bundle.py <solution-root>
```

When a generated stack exists, also run `podman compose config` from `<solution-root>/stack` and record the result in the bundle validation evidence.

## Release Checklist

- `VERSION`, `README.md`, `CHANGELOG.md`, and `NOTICE` mention the same current version
- `scripts/check_skill_package.py` passes
- unit tests pass
- no transient caches or bytecode files exist under the skill folder
- bundled helper snapshots are intentional and still referenced
- deploy archive expands to a top-level `livestacks-orchestrator/` folder
- `livestacks-orchestrator.update.json` has the correct `archive_url`, `archive_sha256`, `content_hash`, `version`, and validation command
- `ready-to-deploy-archive/livestacks-orchestrator.zip` matches the manifest

## Common Pitfalls

- updating `SKILL.md` without updating the public README, changelog, support notes, or version surfaces when release behavior changes
- trusting skill-cleaner dry-run removals without checking bundled helper indexes and references
- using delegated agents without explicit user permission
- treating sparse briefs as permission to ship generic dashboards instead of capability-led operator applications
- calling mock-backed or in-memory runtime behavior production-ready
- letting generated guide scenes drift from the real running app
- refreshing the archive without refreshing manifest hashes

## Expected Output From Codex

- normalized business input and working PRD
- visible role ledger and specialist plan
- one recommended LiveStacks package, not parallel unresolved options
- Podman-first `stack/`, Oracle-first `database/`, ORDS-mediated app APIs, app UI, docs, and sibling `guide/`
- validation output from scaffold-marker, semantic validation, grading, and package checks
- clear blockers when the bundle is not yet production-ready
