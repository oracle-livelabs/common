Skill review

What the LiveLabs Author skill does, and how it works end to end.

The livelabs-author skill turns rough ideas, source documents, blogs, product pages, diagrams, sample SQL, notebooks, slide decks, and provided screenshots into a structured Oracle LiveLabs workshop package. It does more than write markdown: it helps choose an authoring mode, scaffold the workshop, create lab stubs, decide how FreeSQL should appear, validate structure, tighten prose, and return a delivery summary with traceability and QA.

Primary goal

Author

Build or update LiveLabs workshops from source material rather than from a fully formed prior workshop.

Operating style

Self-contained

Bundled references, templates, and helper scripts keep the skill usable even when downloaded alone.

Quality posture

Validator-aware

Targets LiveLabs markdown constraints, manifests, traceability, prose tightening, and QA delivery.

Best use

Workshop production

Use it when you need a real workshop path, not just brainstorming or generic summarization.

Input + structure + evidence + QA + delivery.

Important boundary

Version 26.5.2 intentionally removes screenshot capture and industry conversion responsibilities from the author skill. Use dedicated skills for those tasks when needed.

Skill purpose

A workshop production system, not just a markdown generator.

The skill creates or updates Oracle LiveLabs workshops from source-only inputs. A writable destination folder and credible source material are enough; a GitHub repository, existing workshop repository, or external sample checkout is not required.

Input sources

Starts with real evidence

Ideas, blogs, product docs, web pages, diagrams, commands, screenshots supplied by the user, notebooks, decks, PowerPoint files, and sample SQL can all be used as source material.

Workshop blueprint

Turns source into a learner journey

The planning pass identifies learner outcomes, audience assumptions, environment needs, target duration, lab count, short and long descriptions, outline, ordered labs, and FreeSQL opportunities.

Publish posture

Moves toward review-ready output

The workflow builds structure, authors labs, validates markdown, tightens language, checks renderer-sensitive patterns, and packages the result with QA status and unresolved SME gaps.

Core idea

LiveLabs authoring succeeds when content is not only written, but structured, traceable, renderer-compatible, and easy for a reviewer to validate. This skill makes those steps explicit.

End-to-end flow

The eight-step workflow.

The authoring process is intentionally staged. Scope and evidence come first, then structure, content, FreeSQL choices, manifest coverage, prose cleanup, validation, and final handoff.

01Translate sourceConfirm duration, lab count, Oracle validation needs, audience, outcomes, prerequisites, descriptions, outline, tasks, and FreeSQL opportunities.
02Create skeletonUse the scaffold helper to generate the LiveLabs root layout, variant folders, loader-based index files, and starter manifests.
03Add lab stubsCreate lab folders, image folders, required sections, and metadata stubs before drafting detailed instructions.
04Author labsWrite task-focused markdown with required headings, objectives, estimated time, meaningful alt text, and acknowledgements.
05Add FreeSQLUse runnable SQL, PL/SQL, Quick SQL, or JavaScript only when it materially improves the learner experience.
06Build manifestsRender or refresh manifest.json so the variant tutorial order and lab titles match the final workshop plan.
07Tighten proseApply concise, factual writing guidance and remove filler, vague claims, and inflated AI-generated language.
08Validate and deliverRun QA, repair failures in priority order, self-grade the changed files, and return the delivery contract.
Modes

Choose the authoring depth before drafting.

The mode controls how complete and polished the output should be. Choosing it early prevents a quick skeleton from being mistaken for publish-ready material.

ModeBest fitExpected result
draftFastest path from source to a working markdown skeleton.Initial workshop structure, starter lab content, visible gaps, and lower polish.
publish-readyComplete workshop production with validation and cleanup.Complete markdown, manifest coverage, FreeSQL decisions when relevant, QA pass, repairs, and delivery summary.
how-to-guideA procedural workshop that teaches authors or operators how to build or publish something.Clear steps, checkpoints, operational guidance, and author-oriented explanations.
fastlabNarrow scope, fewer labs, tight setup, and short runtime.Compressed lab flow with minimal setup and focused hands-on tasks.
Workshop structure

It scaffolds a real LiveLabs layout.

The skill expects lower-case folder and file names, loader-based index files, manifest coverage, lab folders, image folders, and reusable metadata that can be pasted into LiveLabs workshop details.

Root

Workshop root

Contains the overall workshop structure, support files, details stub, traceability stub, and variant folders.

Variant

workshops/<variant>

Includes the loader-based index.html and manifest.json for the intended tutorial order.

Labs

Lab folders

Each lab has markdown, an images folder, required sections, and task-oriented content.

Metadata

Details and traceability

Short description, long description, outline, prerequisites, and source mapping are captured for review and reuse.

Markdown rules

The lab markdown has a strict section contract.

The skill writes for the LiveLabs renderer, not generic GitHub markdown. It prioritizes predictable headers, valid task formatting, safe image handling, and clean learner instructions.

One H1 firstEach lab starts with one H1 as the first non-empty line.
Required sectionsEvery lab includes Introduction, Objectives, an estimated time field, and Acknowledgements.
Task headersGuided steps use ## Task ...: ... headers so the flow is renderer-friendly.
Nested indentationNested text, images, lists, and code blocks inside numbered steps are indented by four spaces.
Image alt textEvery image receives meaningful alt text and a valid path.
No invalid anchorsThe workflow avoids HTML anchors and embed syntax that breaks LiveLabs rendering.
Factual introductionsIntroductions focus on the learner problem, environment, and outcome, not the prompt or source article.
Concise languageThe skill cuts filler early and rewrites weak or inflated generated prose before delivery.
FreeSQL decisions

FreeSQL appears only when it helps the learner.

The skill can add runnable FreeSQL content, but it still chooses the right pattern. A plain code block, share link, button, or embedded editor should match the workshop goal and renderer constraints.

Use FreeSQL for

Runnable SQL tasks

SQL, PL/SQL, Quick SQL, or JavaScript examples that materially help the learner can use FreeSQL affordances.

Default when embedded

Iframe pattern

If the user asks for inline, embedded, or in-frame FreeSQL and raw HTML is allowed, use the canonical freesql-embed iframe pattern.

Do not overpromise

Embedding caveat

If FreeSQL does not support a task, such as vector embedding generation, the skill should warn and avoid presenting FreeSQL as the runnable environment.

Validation and repair

QA is ordered so the biggest failures are fixed first.

The skill does not treat validation as an afterthought. It runs the QA wrapper, optionally targets edited files for faster iteration, and repairs failures in a specific priority order.

# Full workshop QA
bash scripts/run_workshop_qa.sh <workshop-root>

# Faster iteration on edited markdown only
bash scripts/run_workshop_qa.sh <workshop-root> --files introduction/introduction.md lab-a/lab-a.md
1. Structure firstFix missing required sections, headers, task formats, and ordered-list indentation before polishing content.
2. Renderer-sensitive itemsRepair image paths, alt text, naming, copy block balance, FreeSQL embeds, time fields, and acknowledgements.
3. Content qualityRemove wordy prose, unsupported claims, and vague AI-generated language.
4. Preview failuresResolve bad loader pages, missing local assets, and broken relative paths that cause local preview issues.
