# Lab 5: Run Live Prep Checks

## Introduction

Use this lab before attendees start hands-on work. Confirm the event path, delivery roles, launch state, and remaining risks before the session moves forward.

Lab 4 owns attendee preparation. Complete its attendee checklist and preflight email before using this final event-readiness check.

### Objectives

In this lab, you will:

- Confirm that [Lab 4 attendee preparation](?lab=lab-4-send-attendee-preflight-prerequisites) is complete.
- Run the [opening prep check](#legend).
- Verify the event path, roles, support route, and fallback.
- Complete the final author checklist.
- Make the go or no-go decision.

2. Resolve every unchecked required item before the event where possible. For an accepted risk, record the owner, fallback, and decision time in the runbook.

## Task 2: Run the Opening Prep Check

1. The lead facilitator states:

    ```text
    We will spend the first few minutes confirming access before we start the hands-on work.
    Please confirm in chat when you see [expected first screen].
    If you hit a blocker, tell us which screen you see and which access path you used.
    ```

2. The screen driver shows the expected path.

3. The chat/support owner records blockers.

4. The lead facilitator confirms whether the group can begin.

## Task 3: Confirm the Event Operations Package

1. Check the event operations package before the event starts.

    | Operations Item | What to Verify | Complete |
    | --- | --- | --- |
    | Event path | Event code, event page, approved link, and QR code work. | <input type="checkbox" aria-label="Event path complete"> |
    | Access path | The team supports one path: sandbox, own tenancy, Secure Desktop, or early-start lab. | <input type="checkbox" aria-label="Access path complete"> |
    | Facilitation runbook | The facilitator, screen driver, chat/support owner, SME, and event coordinator roles are clear. | <input type="checkbox" aria-label="Runbook complete"> |
    | Support and fallback | The team knows when to pause, when to use breakout support, and which fallback to use. | <input type="checkbox" aria-label="Support and fallback complete"> |
    | Attendee preparation | The [Lab 4 checklist and preflight message](?lab=lab-4-send-attendee-preflight-prerequisites) are complete. | <input type="checkbox" aria-label="Attendee preparation complete"> |

2. Update the runbook if any event-operations item changed after the dry run.

3. Keep the verified LiveLab URL, event code, first-screen note, and support route visible to the delivery team.

2. Resolve every unchecked required item before the customer session when possible.

3. If an item must remain unchecked, name the risk and the person who will track it.

## Task 5: Confirm Go or No-Go

1. Confirm the final event state.

    | State | Use When | Next Step |
    | --- | --- | --- |
    | Ready | All required checks are complete. | Start the event as planned. |
    | Ready with risk | One non-blocking item remains. | Name the risk and keep the backup path ready. |
    | Not ready | A required event path or delivery role is missing. | Pause launch and resolve the blocker before attendees begin. |

2. Tell the delivery team which state applies.

3. Keep Lab 6 open during the event.

4. Open the dedicated checklist when the delivery team needs an interactive or paper copy.

    [Open the printable readiness checklist](../../printable-readiness-checklist.html)

## Legend

| Term | Meaning | Why It Matters |
| --- | --- | --- |
| Opening prep check | First live check that confirms attendees reached the expected screen. | Catches access issues before hands-on work starts. |
| Event operations package | Event path, runbook, roles, support route, and fallback needed for delivery. | Keeps final readiness separate from attendee preparation in Lab 4. |
| Runbook | Event flow, roles, handoffs, support route, and backup plan. | Keeps the delivery team aligned during the event. |
